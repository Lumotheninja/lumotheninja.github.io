Learning Go - week 1

29 Aug 2020 on Go

The new language with an adorable mascot

Go is a fairly new language, created in 2009 by Google and version 1.0 was released in 2012. According to Wikipedia, the designers wanted to address criticism of other languages in use at Google, but keep their useful characteristics:

  • static typing and run-time efficiency (like C++),
  • readability and usability (like Python or JavaScript)
  • high-performance networking and multiprocessing.

The designers were primarily motivated by their shared dislike of C++.

Well, I need to admit - my exposure to the C-family of programming language is very limited, since I mainly program in Python and Java, so I have no clue what is so digusting about C++. But the stated characteristics seem promising - Python’s simplicity is cool, static typing is a necessary evil and multiprocessing is cool as well.

I took my first lessons in Go in the official Tour of Go and then steadily progressed to solving Leetcode problems in Go. I believe that solving Leetcode problems is the best way to learn the syntax since it forces you to do basic programmming acrobatics like iterating through the digits of an int or reversing an array or implement sort etc. Since there are many expert materials on learning Go, I am not going to explain Go. Instead, I am going to talk about what tripped me up when first trying to code in Go.

Pointer type and dereference operation are different

This is a function that takes in a pointer type

func doesSomething (*int arg) bool

While this is a dereferenced pointer

value := 1
pointer := &value //pointer
x := *pointer //value

In the first example, *something actually means this is a pointer type, but in the second example, *something is an operation to dereference a pointer to its value. This is something that has confused me as I randomly switch between & and * to try to find the correct operator. It took a while for my brain to comprehend that one is a type declaration while the other is a dereference operator.

String conversion is not that straightforward

Having only experience in Java and Python, I have no knowledge of the C-family functions Atoi and Itoa. So imagine my suprise when string(123) yielded “{“ instead of “123”. So how do we get a string “123”? we need to use the C family function Itoa (integer to Ascii)

fmt.Println(strconv.Itoa(123)) // prints 123
fmt.Println(string(123)) // prints {

This is confusing to me since the default type conversion is to treat the int as a unicode point which is different from the Python implementation. If you want to convert the other way from string to int, you have to use Atoi. Even weirder are string iterations.

Say hello to runes

In Python, there is no distinction between chars and str. A single character is basically a string, but in Go, you have rune and string. Rune is just an alias for int32 (4 bytes) and represents the Unicode int representation. Whenever you loop a string, you will have to loop through the runes in the string, such as below:

for i, ch := range "Japan 日本" {
    fmt.Printf("%d:%q ", i, ch)
}
// Output: 0:'J' 1:'a' 2:'p' 3:'a' 4:'n' 5:' ' 6:'日' 9:'本'

Although runes represent the individual characters in strings, the length of a string is represented in bytes, which explains the strange indexing above (6:’日’ 9:’本’). Meaning teh character ‘日’ itself is 3 bytes long

Channels as a single threaded queue?

I was doing a tree traversal question on Leetcode that requires a queue when I suddenly had the idea to try use channels as a queue. After all, channel also works in a FIFO way similar to a queue right? This was the code I have written to simulate a single threaded queue:

type TreeDepth struct {
    Tree *TreeNode
    Depth int
}

func addToQueue(t TreeDepth, queue chan TreeDepth) {
    if left := t.Tree.Left; left != nil{
        queue <- TreeDepth{left, t.Depth+1}
    }
    if right := t.Tree.Right; right != nil {
        queue <- TreeDepth{right, t.Depth+1}
    }
}
func maxDepth(root *TreeNode) int {
    queue := make(chan TreeDepth)
    maxDepth := 0
    go addToQueue(TreeDepth{root, 1}, queue)
    for {
        select {
            case t :=  <-queue:
                fmt.Println(t.Tree.Val, t.Depth)
                if t.Depth > maxDepth {
                    maxDepth = t.Depth
                }
                go addToQueue(t, queue)

            default:
                return maxDepth
        }
    }
}

However, the problem is that the function that pushes to the channel cannot be run in the main thread or it blocks. On the other hand, if you run it in a separate thread, you cannot determine that the push has already happened when you read. Thus, the biggest difference of a channel and a single threaded queue is this: it blocks when it pushes or reads. This is true to its design to synchronize between threads, and should not be used as an actual queue in the single threaded sense.
